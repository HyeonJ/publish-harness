# Home G1 Dogfooding 요약


## 한 줄 요약

처음에는 하네스가 "화면이 다르다" 정도만 말해줬다.  
dogfooding을 거치면서 이제는 "어느 구역이 왜 다른지, 다음에 무엇을 봐야
하는지"까지 더 잘 알려주게 됐다.

## 왜 이 작업을 했나

Figma 디자인을 React 화면으로 옮긴 뒤, 하네스는 화면이 디자인과 얼마나
다른지 검사한다.

그런데 실제 작업을 해보니 단순히 "실패"라고만 알려주는 것으로는 부족했다.
작업자는 다음을 알아야 했다.

- 아직 중요한 요소가 화면에 연결되지 않았는지
- 연결은 됐지만 위치가 틀린 것인지
- 섹션 전체가 밀린 것인지
- 텍스트 크기 문제인지
- 이미지 순서나 색상이 다른 것인지
- 지금 고칠 문제가 맞는지, 아니면 사람의 판단이 필요한지

이번 dogfooding은 이 질문들에 하네스가 더 잘 답하도록 만드는 과정이었다.

## 가장 크게 좋아진 점

### 1. 덜 중요한 누락 때문에 막히지 않게 됨

기존에는 필수 요소가 모두 연결돼 있어도, 덜 중요한 장식 요소가 빠졌다는
이유로 검사가 막힐 수 있었다.

이제는 필수 요소만 제대로 연결되면 다음 단계로 넘어가고, 덜 중요한 누락은
"참고 사항"으로만 보여준다.

### 2. 같은 화면 요소에 여러 Figma 이름을 붙일 수 있게 됨

Figma에는 같은 박스가 여러 레이어로 나뉘어 있는 경우가 많다.

예를 들면 하나의 footer 배경인데 Figma에는 section, rectangle, background
같은 이름이 여러 개 붙어 있을 수 있다.

이제 하네스는 이런 경우 하나의 실제 화면 요소에 여러 Figma 이름을 함께
연결할 수 있다. 그래서 숨겨진 가짜 요소를 만들 필요가 줄었다.

### 3. "아래쪽이 밀림"의 진짜 원인을 더 잘 찾게 됨

처음에는 footer가 아래로 밀렸다고만 나왔다.

하지만 실제 원인은 footer가 아니라 그 위의 섹션들이었다.

- 어떤 섹션은 너무 높았다.
- 어떤 섹션 사이 간격이 잘못됐다.
- 같은 상품 행이 반복해서 같은 만큼 커져 있었다.

이제 하네스는 이런 문제를 나눠서 보여준다.

- 섹션 자체 높이 문제
- 섹션 사이 간격 문제
- 앞쪽 문제 때문에 뒤쪽이 밀린 문제

### 4. 반복되는 문제를 묶어서 보여주게 됨

상품 행 3개가 모두 같은 이유로 높이가 120px씩 커져 있었다.

예전에는 세 개의 별도 문제처럼 보였다.  
이제는 "같은 패턴이 반복되고 있다"고 묶어서 보여준다.

이 덕분에 하나의 공통 컴포넌트나 스타일을 먼저 의심할 수 있다.

### 5. 텍스트 문제와 연결 문제를 구분하게 됨

어떤 텍스트가 디자인보다 너무 넓게 측정됐다.

처음에는 글자 크기나 줄간격 문제처럼 보였다.  
하지만 실제로는 텍스트 자체가 아니라 넓은 부모 박스에 잘못 연결된 문제였다.

이제 하네스는 이런 경우를 더 잘 구분한다.

- 글자 크기를 고쳐야 하는 문제
- 줄바꿈을 고쳐야 하는 문제
- 텍스트 위치를 고쳐야 하는 문제
- Figma 이름이 잘못 연결된 문제
- 너무 큰 박스에 연결된 문제

### 6. Figma 레이어 이름 때문에 헷갈리는 일이 줄어듦

Figma 레이어 이름이 실제 문구와 다를 때가 있었다.

예:

- `watermellon mania`처럼 오타가 있는 이름
- `logoname`처럼 실제 문구가 아닌 역할 이름
- `a paragraph...`처럼 임시 이름

이제 하네스는 이런 항목을 바로 앱 수정 대상으로 보지 않고,
"확신이 낮은 참고 항목"으로 분리한다.

### 7. 작은 수정으로 안 되는 섹션을 구분하게 됨

어떤 섹션은 단순히 여백 몇 픽셀을 고쳐서 해결되지 않았다.

Figma는 자유롭게 배치된 레이아웃인데, React 화면은 규칙적인 카드 그리드로
만들어져 있었기 때문이다.

이제 하네스는 이런 경우를 다음처럼 판단할 수 있게 됐다.

- 계속 조금씩 맞출 문제인지
- 섹션 구조를 다시 짜야 하는지
- 사람의 결정이 필요한지

### 8. 전체 화면 차이가 큰 구역을 찾아주게 됨

요소 위치 문제가 어느 정도 정리된 뒤에도 전체 이미지 차이는 컸다.

이때 하네스는 이제 어느 섹션이 전체 차이를 가장 크게 만들고 있는지 보여준다.

예를 들어 dogfooding 중에는 다음 문제를 찾는 데 도움이 됐다.

- footer 배경색이 다름
- bundle 섹션 배경색이 다름
- benefit 이미지 순서가 다름
- sale banner 안의 문구 순서가 다름

## 하네스가 이제 더 잘 말해주는 것

하네스는 단순히 실패 여부만 말하는 대신, 다음 작업을 더 구체적으로 안내한다.

예:

- 필수 요소부터 연결하세요.
- 섹션 사이 간격을 확인하세요.
- 반복되는 카드/행 높이를 확인하세요.
- 텍스트가 너무 큰 박스에 연결되어 있습니다.
- 이건 텍스트 크기 문제가 아니라 연결 문제입니다.
- 이 섹션은 작은 튜닝보다 구조 변경이 필요해 보입니다.
- 이제는 위치보다 이미지/색상 차이를 봐야 합니다.

## 이번 작업으로 바뀐 주요 파일

주로 다음 파일들이 개선됐다.

- `scripts/_lib/playwright-stable.mjs`
- `docs/codex-section-worker.md`

## 아직 더 좋아져야 할 점

아직 하네스가 완벽한 것은 아니다.

특히 다음 부분은 더 개선하면 좋다.

- 문제가 있는 섹션의 이미지 후보를 더 정확히 알려주기
- 비교용 crop 이미지를 자동으로 만들어주기
- 어떤 DOM 요소가 측정됐는지 selector/class 보여주기
- 이미지 순서 문제와 crop 문제를 더 명확히 구분하기
- "다음 작업 추천"에 신뢰도 점수 붙이기

## 결론

이번 dogfooding 전에는 하네스가 주로 이렇게 말하는 수준이었다.

```text
디자인과 화면이 다릅니다.
```

dogfooding 후에는 훨씬 더 구체적으로 말할 수 있게 됐다.

```text
필수 요소는 연결됐습니다.
footer가 밀렸지만 footer 자체 문제가 아니라 위 섹션의 높이/간격 문제입니다.
이 텍스트는 글자 크기 문제가 아니라 잘못된 박스에 연결된 문제입니다.
이제 남은 큰 차이는 section-3의 이미지/순서/crop 문제로 보입니다.
```

즉, 하네스가 단순한 검사 도구에서 작업자가 다음 판단을 할 수 있게 돕는
진단 도구에 가까워졌다.
